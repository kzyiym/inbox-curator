# Runtime Validation Report

## 概要と目的
本レポートは、**Phase 1: 実機 validation と runtime hardening** に基づき、プラグインが Obsidian 上で安全かつ期待通りに動作するかを検証するためのドキュメントです。
エージェントによるサンドボックス環境でのコード監査・自動テストで検証された領域と、実 Obsidian アプリ環境で人間が手動で実測・確認する必要がある領域（未実測項目）を明確にし、検証用のチェックリストを提供します。

---

## 1. 検証ステータス分類

### A. 自動テストまたはコード監査で検証済みの項目 (Verified by Test/Audit)
以下の挙動は、Vitestによるユニットテストおよびモック、あるいは静的コード監査によって動作が検証されています。
* **フロントマターの処理 (`frontmatter.test.ts`)**:
  * メタデータ（ハッシュ、URL、添付ファイル数など）の追加・更新・削除処理。
  * `ai_review_*` 以外の既存フロントマターが保護され、壊されないこと。
* **添付ファイルのパースと分類 (`attachmentContext.test.ts`)**:
  * Wikilink および Markdown リンク/埋め込みの検出。
  * 重複する添付ファイルパスのカウント防止。
  * 拡張子に基づくメディア種別の自動分類。
* **URLコンテンツの抽出とメタデータ取得 (`urlExtraction.test.ts`, `reviewPipeline.test.ts`)**:
  * URLのみのノート判定ロジック。
  * HTMLメタデータのパースとフォールバック。
  * テキストの抽出および最大文字数キャップの動作。
* **並列処理キューの基本動作 (`reviewQueue.test.ts`)**:
  * `maxConcurrentJobs` による並列度制限。
  * 同一ファイルパスのジョブの二重実行排除（重複 enqueue 拒否）。

### B. 実機で検証すべき未実測項目 (Unverified Runtime Behavior)
以下の項目は、実 Obsidian のイベントループ、実際の Vault ファイルシステム、非同期API通信、デスクトップ/モバイル特有の挙動に依存するため、実機での検証が必要です。
* **実 Obsidian Vault のファイルイベント (`create` / `modify`)**:
  - `vault.on('create')` や `vault.on('modify')` イベントがファイルシステム変更時に正しく発火し、意図しない無限ループ（レビュー結果の書き込みで `modify` が走るなど）が発生しないか。
* **Polling Fallback の挙動**:
  - バックグラウンドで定期的にポーリングが走り、イベント漏れノートを拾えるか。
  - ポーリングと手動/自動キューイングの間で、二重 enqueue や競合が発生しないか。
* **Unload / Reload のクリーンアップ**:
  - プラグインが無効化 (`onunload`) されたとき、ポーリングタイマー、ウォッチのデバウンスタイマー、キューの処理、APIリクエストのリミッターが完全に停止・解放されるか。
* **SecretStorage の実挙動**:
  - APIキーが Obsidian の提供する SecretStorage に正常に保存・復元されるか。
  - APIキー設定時に UI マスクが正常に行われるか。
* **Notice / Status bar のUX**:
  - レビュー実行中およびエラー発生時に、トースト通知 (Notice) やステータスバーが崩れずに表示されるか。
* **外部 API / ネットワーク通信**:
  - 実際の OpenAI 互換プロバイダーに対してリクエストが飛び、正常なレビューノートが出力されるか。
  - レートリミット（RPM）やバックオフ再試行が、本物のネットワークエラー時にも適切に機能するか。

## 2. 実機検証チェックリスト (Manual Verification Checklist)

実機で以下のシナリオを実行し、挙動に問題がないことを確認してください。

### 🟩 シナリオ 1: 基本コマンドと API 連携
- [ ] **コマンド実行**: `Review current note` を実行し、現在開いているノートがレビューされること。
- [ ] **Notice / Status bar 表示**: レビュー中に「Reviewing current note...」が表示され、完了後に「Review completed」と通知されること。二重に実行しようとした際「Review already in progress」と防がれること。
- [ ] **API接続と出力**: レビュー結果が設定された出力フォルダ内に `*.ai-review.md` として書き出されること。
- [ ] **フロントマター書き戻し**: 元ノートに `ai_review_status` などのメタデータが正しく書き込まれること。元の文章が破壊されないこと。
- [ ] **SecretStorage 動作**: APIキーを設定してプラグインをリロードした後、再度レビューを実行し、キーが保持されていることを確認。

### 🟩 シナリオ 2: フォルダ一括処理 (Process watched folder)
- [ ] **一括処理の制限とスキップ**: 監視フォルダ内に複数の新規/更新ノートを置き、`Process watched folder` を実行。`Max notes per run` の制限数だけ処理され、完了済みの（ハッシュ値が一致する）ノートはスキップされること。
- [ ] **キャンセルとレートリミット**: 処理中に Obsidian の Notice やログを確認し、設定した `Requests per minute` / `Delay between requests` に応じた適切なウェイトが入ることを確認。

### 🟩 シナリオ 3: 自動監視 (Automatic watching)
- [ ] **自動ウォッチ動作**: `Automatic watching`, `Auto-review on create/modify` を ON にする。
- [ ] **新規作成時の挙動**: 監視フォルダ内に新しい Markdown ノートを作成すると、デバウンス時間（`Watch debounce`）経過後に自動でキューに追加され、レビューが開始されること。
- [ ] **更新時の挙動**: 既存の監視対象ノートを編集し、しばらく待つと自動でハッシュ値が更新され、再レビューされること。
- [ ] **無限ループ防止**: レビュー完了時のフロントマター書き込みによって自作自受の modify イベントが発生し、レビューの無限ループが発生しないこと（ソースハッシュ計算時に `ai_review_*` フロントマターを除外しているため、ハッシュは変化しないはず）。

### 🟩 シナリオ 4: ポーリング (Polling fallback)
- [ ] **ポーリング挙動**: `Polling fallback` を ON にする。
- [ ] **変更ノートの検知**: 自動ウォッチを OFF にした状態で監視フォルダ内のノートを編集し、ポーリング間隔経過後に自動で検知・レビューされること。
- [ ] **二重実行の排除**: ポーリング中に手動レビューや他のイベントが重なっても、同一ノートに対するジョブが二重に enqueue されないこと。

### 🟩 シナリオ 5: クリーンアップ (Unload/Reload)
- [ ] **プラグインの無効化**: 設定画面からプラグインを無効化、または Obsidian 自体を閉じる。
- [ ] **リソースの解放**: タイマー（デバウンス、ポーリング）がクリアされ、進行中のキューやAPI接続が安全にキャンセルまたは中断されること。再起動後に予期せぬ実行（再実行地獄）が始まらないこと。

## 3. キューの永続化 (Queue Persistence) に関する設計判断

### 💡 判断: インメモリキューを維持し、キュー自体の永続化は行わない。
* **理由 1: 元ノートのフロントマターが「状態ストア」として機能しているため**
  本プラグインは処理完了時に `ai_review_source_hash` を元のノートに書き込みます。これにより、アプリのクラッシュやリロードが発生してキューが消失しても、再度スキャン（Process watched folder や polling/watch イベント）が走った際に、ハッシュ値の一致によって不要なレビューを瞬時にスキップできます。つまり、実質的に Vault 自体が完了状態を永続化しているため、キューをファイルやデータベースに保存する必要性が極めて低いです。
* **理由 2: 予期せぬ「再実行地獄」と競合の回避**
  Obsidian はリロードや終了が頻繁に起こるデスクトップ/モバイルアプリです。キュー自体をローカルストレージやファイルに永続化した場合、起動時に古いペンディングジョブが一気に走り出すことで、API利用料の急増や、ファイル更新の競合（writebackの競合）を起こす危険性が高まります。起動時は空のキューから開始し、ユーザーの明示的な操作や新しいファイルイベントから再度キューイングする方が、極めて安全で予測可能（Predictable）です。
